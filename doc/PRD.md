# 产品需求文档 (PRD) - 有钱么 (YouQianMe)

## 1. 文档综述

### 1.1 项目背景
“有钱么 (YouQianMe)”是一款面向个人的资产负债管理应用。目标用户希望用更少的步骤，获得清晰、可信的财务全景：当前净资产、变动原因、收益来源、以及 FIRE 目标进度。

### 1.2 产品目标
- 核心价值：极简、高效、可追溯的个人资产管理体验，避免传统记账软件的功能冗余与数据噪声。
- 关键体验：
  - 一屏看清：总资产、总负债、净资产、收益与趋势
  - 数据可信：每次变动可追溯（时间轴/台账），关键计算口径透明
  - 多资产类型：股票/现金/定期/实物资产等统一口径汇总
  - 可行动：支持转账台账、定期兑付、负债还款记录等“会影响余额”的操作

### 1.3 目标用户与场景
- FIRE 规划者：关注净资产趋势、可计入 FIRE 的资产范围、目标达成进度。
- 多账号投资者：一个人多券商/多币种，想在一个面板看到总览与分账号汇总。
- 轻量管理者：只想记录关键资产与负债，不想记录每笔消费。

### 1.4 成功指标（MVP 口径）
- 用户可在 2 分钟内完成：添加 3 个资产 + 1 个负债 + 设置本位币 + 看到净资产与趋势。
- 任意一次资产/负债/台账操作都能在时间轴中找到对应记录（含金额前后对比/变动 delta）。
- 同一份数据在总览、列表、趋势图、FIRE 状态中口径一致（汇率/计入范围/币种折算一致）。

### 1.5 术语与口径
- 本位币：用于全局汇总展示的主币种（例如 CNY）。
- 折算：非本位币资产按汇率折算到本位币后再参与汇总。
- 市值/估值：资产在当前时点的价值；可来自手动录入、市场数据、或固定估算。
- P&L/收益：
  - 市场收益：资产价格变动带来的收益（例如股票涨跌）
  - 活动变动：人为增减仓、转入转出导致的净资产变化
  - 利息收益：定期存款/负债利息等估算或记录
- 计入 FIRE：资产是否被计入 FIRE 可用资产（用于 FIRE 目标进度计算）。

## 2. 功能架构与导航

### 2.1 模块列表（Web/Mobile 通用）
- 总览（Dashboard）

## 3. 详细功能需求

### 3.1 总览（Dashboard）

#### 3.1.1 用户目标
- 快速判断：当前净资产与趋势是否健康
- 快速解释：最近变化来自哪里（资产涨跌、转入转出、负债变化）
- 快速行动：添加资产/负债、修复缺失的市场数据、查看最近记录

#### 3.1.2 核心组件与规则
- 核心指标卡片：
  - 总资产（折算到本位币后的资产合计）
  - 总负债（折算到本位币后的负债合计）
  - 净资产（总资产 - 总负债）
  - 收益（可配置为：累计收益 / 本月收益 / 当日收益；MVP 默认展示本月合计与当日）
- 资产/负债结构摘要（可选）：
  - 按资产类型聚合（股票/现金/定期/实物等）
  - 按币种聚合（CNY/USD 等）
  - 按账户聚合（券商/银行卡等）
- 净资产趋势图：
  - 维度：按日（或按记录发生日）聚合净资产
  - 交互：Hover 显示日期与当日净资产；点击可跳转到当日的收益/时间轴
  - 范围：全部 / 近 30 天 / 近 90 天 / 自定义（MVP 可先做全部/近 30 天）
- 最近动态（时间轴摘要）：
  - 展示最近 N 条（默认 10）
  - 支持筛选：全部 / 资产 / 负债
  - 点击一条记录进入“时间轴详情页”定位到该条
- 市场数据缺失提示：
  - 当存在需要折算但缺少汇率/价格时，提示“缺少市场数据：USD/CNY、XAU/CNY 等”
  - 提供入口跳转到市场数据管理页，或在 Dashboard 内直接快速补录

#### 3.1.3 异常与空状态
- 首次使用无数据：展示引导态（添加资产/负债、设置本位币）
- 缺少市场数据：明确列出缺失项，并保证总览数值仍可展示（对缺失项采用 0 或“不可计算”标识，需在 UI 上可解释）

#### 3.1.4 用户故事与验收标准（Given/When/Then）
- 用户故事 D-1：作为首次使用用户，我希望系统引导我完成最小配置，从而快速看到净资产。
  - Given：系统中没有任何资产与负债
  - When：我进入 Dashboard
  - Then：展示空状态引导（添加资产、添加负债、设置本位币），且不会显示误导性的 0 值趋势图
- 用户故事 D-2：作为用户，我希望看到“可解释”的总览数字，从而知道缺少什么数据会影响计算。
  - Given：存在 USD 资产，但系统缺少 USD/CNY 汇率
  - When：我进入 Dashboard
  - Then：页面显示“缺少市场数据：USD/CNY”，并提供跳转/快捷补录入口；同时汇总数值需明确标记为“部分可计算”或对缺失项有可解释处理
- 用户故事 D-3：作为用户，我希望从 Dashboard 快速跳到具体变动记录，从而定位净资产变化原因。
  - Given：存在最近动态列表
  - When：我点击某条最近动态
  - Then：打开时间轴页面并定位到该条记录（或在详情中展示该条完整信息）
- 用户故事 D-4：作为用户，我希望趋势图提供有效交互，从而快速定位异常日期。
  - Given：趋势图存在至少 7 天数据
  - When：我悬停/触摸某个点
  - Then：显示该日净资产与日期；不会出现 NaN/Infinity；缺失数据点以断点或空洞明确表达

### 3.2 时间轴（Activity Timeline）

#### 3.2.1 用户目标
- 回溯每次资产/负债变化的原因与幅度
- 快速筛选：只看资产或只看负债的变化

#### 3.2.2 数据结构（面向展示）
- 每条记录包含：
  - 时间（精确到秒）
  - 对象类型：资产/负债/交易/设置（MVP 可先做资产/负债）
  - 行为：新增/更新/删除/兑付/还款/转账（按实现逐步补齐）
  - 对象名称（目标名称）
  - 变动金额 delta（用于快速判断影响方向）
  - 金额前后对比（old → new，适用于更新类操作）
  - 备注（可选，解释本次调整的原因）

#### 3.2.3 交互与规则
- 筛选：全部 / 资产 / 负债（MVP）
- 分页/加载更多：默认 50 条，支持“加载更多”最多到 200 条（或无限滚动）
- 点击记录：可展开详情（展示更多字段，如币种、关联账户、关联交易）

#### 3.2.4 用户故事与验收标准（Given/When/Then）
- 用户故事 A-1：作为用户，我希望资产/负债的增删改都会进入时间轴，从而可回溯。
  - Given：我有一条资产或负债记录
  - When：我新增/编辑/删除该记录
  - Then：时间轴出现对应 CREATE/UPDATE/DELETE 记录，且能看到对象名称与变动金额（更新时可见 old → new）
- 用户故事 A-2：作为用户，我希望筛选能快速生效，从而只看我关心的对象类型。
  - Given：时间轴中同时存在资产与负债记录
  - When：我切换筛选到“资产”
  - Then：列表仅显示资产记录；筛选状态在 UI 上持续可见
- 用户故事 A-3：作为用户，我希望加载更多不会打乱我当前筛选与滚动位置，从而连续阅读。
  - Given：我已选择某个筛选并滚动到列表底部
  - When：我点击“加载更多”
  - Then：新增记录追加到列表底部；筛选不变；不会闪回顶部
- 用户故事 A-4：作为用户，我希望错误态可恢复，从而不会因一次网络/数据库错误卡死页面。
  - Given：加载时间轴数据发生错误
  - When：我刷新或再次触发加载
  - Then：页面能恢复到正常状态（或提示可重试），不会永久停留在 Loading

### 3.3 资产管理（Assets）

#### 3.3.1 用户目标
- 管理所有正向资产（金融资产 + 实物资产）并统一折算汇总
- 对可计入 FIRE 的资产进行标记，以便 FIRE 进度计算

#### 3.3.2 资产类型（建议统一枚举）
- 金融类：STOCK（股票）、OPTION（期权）、CASH（现金/活期）、TIME_DEPOSIT（定期存款）、FUND（基金/理财）
- 实物类：GOLD（黄金）、REAL_ESTATE（房产）、VEHICLE（车辆）、LUXURY（奢侈品）、OTHER（其他）

#### 3.3.3 资产字段（MVP 最小集合）
- 基础信息：
  - 名称（必填）
  - 类型（必填）
  - 币种（默认本位币）
  - 计入 FIRE（默认：金融类是，实物类可选）
  - 关联账户（可选，用于多账号汇总）
- 估值相关（按类型不同启用不同字段）：
  - 金额（现金/定期/其他固定估值）
  - 数量 + 单价（股票/基金/黄金等，单价可来自市场数据或手动）
  - 估值方式：手动 / 市场价 / 固定估算（MVP 可先做手动）
- 元数据：
  - 更新时间（自动）
  - 备注（可选）

#### 3.3.4 列表与详情页展示
- 资产列表：
  - 默认按类型分组或按价值排序（MVP 价值排序）
  - 每行展示：名称、类型、折算后的价值（可在副信息展示原币种金额）
  - 支持快速编辑（点击进入编辑页/弹窗）
- 资产详情（可选）：
  - 展示历史估值变化（若有记录）
  - 展示关联交易（从台账筛选出来）

#### 3.3.5 新增/编辑/删除流程与校验
- 新增：
  - 校验必填字段（名称/类型/币种）
  - 金额/数量/单价必须为非负数，超大值提示二次确认
  - 保存后写入时间轴（CREATE）
- 编辑：
  - 保存后写入时间轴（UPDATE）
  - 若涉及金额变化，时间轴需展示 old → new 以及 delta
- 删除：
  - 二次确认（避免误删）
  - 删除后写入时间轴（DELETE）

#### 3.3.6 定期存款（Time Deposits）专项
- 字段（建议）：
  - 本金、起息日、到期日、年化利率、是否到期自动续存（可选）
- 计算与展示：
  - 显示到期日、预计到期本息（估算）
  - 在总览/收益日历中将利息部分计入“利息收益”
- 到期兑付动作：
  - 用户在资产页触发“兑付”
  - 系统将定期资产状态更新为已兑付（或将金额转入现金资产，取决于台账口径）
  - 写入时间轴（REDEEM）并可生成对应台账记录（推荐）

#### 3.3.7 用户故事与验收标准（Given/When/Then）
- 用户故事 AS-1：作为用户，我希望能新增一笔现金资产，从而把关键现金纳入总资产。
  - Given：我在资产页
  - When：我新增一笔 CNY 现金资产并输入金额
  - Then：资产列表出现该资产；Dashboard 总资产与净资产同步更新；时间轴新增 CREATE 记录
- 用户故事 AS-2：作为多币种用户，我希望看到原币金额与折算金额，从而理解汇总口径。
  - Given：我新增了一笔 USD 资产
  - When：我打开资产列表
  - Then：列表同时展示原币金额与折算后的本位币金额；若缺汇率则提示缺失并可补录
- 用户故事 AS-3：作为用户，我希望编辑资产金额时能看到变更差异，从而确认自己改对了。
  - Given：资产金额为 10,000
  - When：我将金额改为 12,000 并保存
  - Then：时间轴记录展示 10,000 → 12,000 与 delta=+2,000；列表与总览数值同步更新
- 用户故事 AS-4：作为 FIRE 规划者，我希望能控制资产是否计入 FIRE，从而得到更准确的进度。
  - Given：某资产当前计入 FIRE
  - When：我关闭“计入 FIRE”并保存
  - Then：FIRE 页面中的“当前 FIRE 资产/完成率”随之变化；时间轴出现 UPDATE 记录
- 用户故事 AS-5：作为用户，我希望对定期存款进行到期兑付，从而使余额与收益可追溯。
  - Given：存在一笔未兑付的定期存款且已到期（或允许提前兑付）
  - When：我点击“兑付”并确认
  - Then：资产状态/余额按规则变化；时间轴出现 REDEEM 记录；若生成台账记录则可在交易台账中查到

### 3.4 负债管理（Liabilities）

#### 3.4.1 用户目标
- 记录所有负债余额与到期信息
- 记录还款行为与负债余额变化，支持回溯

#### 3.4.2 负债类型与字段（MVP）
- 类型：信用卡、房贷、车贷、消费贷、其他借款
- 字段：
  - 名称（必填）
  - 币种（默认本位币）
  - 当前待还余额（必填，非负）
  - 到期日（可选）
  - 年化利率（可选，用于利息估算）
  - 备注（可选）

#### 3.4.3 列表与交互
- 列表展示：名称、余额、到期日（如有）、折算值（如非本位币）
- 新增/编辑/删除：与资产一致，变更写入时间轴

#### 3.4.4 还款记录（Payments）专项
- 用户可为某一负债新增一条“还款记录”：
  - 字段：日期、金额、备注（可选）、来源资产（可选，用于台账联动）
  - 规则：还款金额必须 > 0 且不应超过当前余额（允许超额但需要二次确认并给出处理方式）
  - 结果：负债余额减少；写入时间轴（PAYMENT/UPDATE）
- 展示：
  - 在负债详情页展示还款历史列表
  - 支持删除某条还款记录（需要回滚余额并写入时间轴）

#### 3.4.5 用户故事与验收标准（Given/When/Then）
- 用户故事 LI-1：作为用户，我希望新增一笔负债，从而把债务纳入净资产计算。
  - Given：我在负债页
  - When：我新增负债并输入余额与到期日（可选）
  - Then：负债列表出现该项；Dashboard 总负债与净资产更新；时间轴新增 CREATE 记录
- 用户故事 LI-2：作为用户，我希望记录一次还款，从而余额减少且可回溯。
  - Given：某负债余额为 5,000
  - When：我新增一条还款记录金额 1,000
  - Then：负债余额变为 4,000；还款历史出现该条；时间轴出现 PAYMENT/UPDATE 记录
- 用户故事 LI-3：作为谨慎用户，我希望系统阻止明显错误输入，从而避免录入负数或异常金额。
  - Given：我正在录入还款金额
  - When：我输入 0 或负数
  - Then：系统阻止提交并提示错误原因
- 用户故事 LI-4：作为用户，我希望能撤销一条还款记录，从而修正误操作。
  - Given：我已添加一条还款记录
  - When：我删除该还款记录并确认
  - Then：负债余额回滚；还款历史删除；时间轴出现对应记录（DELETE 或回滚说明）

### 3.5 交易台账（Transactions / Ledger）

#### 3.5.1 用户目标
- 用结构化台账记录“资产之间的转移/买卖/入金/出金/还款”等影响余额的行为
- 让净资产变化可解释（活动变动 vs 市场收益 vs 利息收益）

#### 3.5.2 交易类型（建议）
- 转账：资产 A → 资产 B（跨币种/跨账户）
- 入金/出金：外部现金流进入或离开系统
- 负债还款：资产减少 + 负债减少（推荐与 Payments 联动）
- 定期兑付：定期资产 → 现金资产（或直接状态变更）

#### 3.5.3 字段与规则（MVP）
- 字段：
  - 日期时间（默认当前）
  - 类型（必填）
  - From（可选，取决于类型）
  - To（可选，取决于类型）
  - 金额与币种（必填）
  - 手续费（可选，默认 0）
  - 备注（可选）
- 规则：
  - 金额必须 > 0
  - From/To 不能相同（转账场景）
  - 保存后对关联资产/负债余额产生一致性变更，并写入时间轴

#### 3.5.4 用户故事与验收标准（Given/When/Then）
- 用户故事 TX-1：作为用户，我希望记录一次转账，从而资产之间的变动可追溯。
  - Given：存在资产 A 与资产 B
  - When：我新增一条转账交易（From=A，To=B，金额>0）并保存
  - Then：交易列表出现该记录；A 余额减少、B 余额增加（或按口径处理）；时间轴出现 TRANSFER 记录
- 用户故事 TX-2：作为用户，我希望台账的变更是原子性的，从而不会出现“只改了一边”的脏数据。
  - Given：保存交易需要同时更新多个对象
  - When：保存过程中发生错误
  - Then：交易与余额都不应产生部分写入；页面提示可重试
- 用户故事 TX-3：作为用户，我希望删除一条交易会回滚其影响，从而修正误记。
  - Given：某交易已影响资产余额
  - When：我删除该交易并确认
  - Then：相关资产/负债余额按相反方向回滚；时间轴出现 DELETE 记录

### 3.6 收益日历（Earnings Calendar）

#### 3.6.1 用户目标
- 用“日历热力图”查看每天收益幅度，定位异常波动日期
- 点击某一天查看收益拆分（活动变动/市场收益/利息等）

#### 3.6.2 页面结构与交互
- 月历视图：
  - 默认显示本月
  - 支持上一月/下一月切换
  - 每个日期格展示颜色深浅代表当日收益绝对值大小（红涨/绿跌/灰接近 0）
- 侧边/下方详情：
  - 点击某天显示该天拆分：total、activity、market、interest（字段可按实现调整）

#### 3.6.3 数据与口径（MVP）
- 请求区间：按月取 start/end 日期范围
- 计算：
  - 当日总收益 = 当日各来源之和
  - 缺失市场数据时：相关来源置为 0 或标记不可计算，并在 UI 给出提示

#### 3.6.4 用户故事与验收标准（Given/When/Then）
- 用户故事 ER-1：作为用户，我希望按月查看收益热力图，从而快速定位异常波动日期。
  - Given：当前展示某个月份
  - When：我点击“上个月/下个月”
  - Then：展示加载态；请求新月份数据；完成后月历与月度合计更新
- 用户故事 ER-2：作为用户，我希望点击某一天查看拆分，从而知道收益来源。
  - Given：月历已加载
  - When：我点击某个日期
  - Then：显示该日 total 与拆分项（activity/market/interest 等）；没有数据的日期展示“无记录/0”
- 用户故事 ER-3：作为用户，我希望缺失市场数据时仍能正常浏览，从而不会阻断复盘。
  - Given：该月存在缺失汇率/价格导致 market 不可计算
  - When：我查看该月日历与某日详情
  - Then：UI 明确提示“部分不可计算”；已可计算部分仍正常展示；提供跳转补录入口

### 3.7 FIRE 目标（FIRE Goal）

#### 3.7.1 用户目标
- 配置 FIRE 参数并看到“目标资产”与当前进度
- 明确哪些资产计入 FIRE，并能通过开关快速调整口径

#### 3.7.2 配置项（建议）
- 年支出（必填）：Annual Expenses
- 安全提取率（必填）：SWR（默认 4%）
- 目标资产 = 年支出 / SWR
- 进度 = 计入 FIRE 的资产折算总额 / 目标资产

#### 3.7.3 展示
- 目标资产、当前 FIRE 资产、完成率、预计差额
- 列出“计入 FIRE 的资产清单”与“未计入的资产清单”（可选）

#### 3.7.4 用户故事与验收标准（Given/When/Then）
- 用户故事 FI-1：作为用户，我希望配置年支出与 SWR，从而获得目标资产与进度。
  - Given：我进入 FIRE 页面
  - When：我输入年支出与 SWR 并保存
  - Then：页面展示目标资产、当前 FIRE 资产、完成率、差额；并在刷新后仍保持该配置
- 用户故事 FI-2：作为用户，我希望调整“计入 FIRE”的资产后进度同步变化，从而可以快速做口径试算。
  - Given：存在若干资产，其中部分计入 FIRE
  - When：我切换某资产的“计入 FIRE”
  - Then：FIRE 页面进度立即反映变化；时间轴记录该变更
- 用户故事 FI-3：作为用户，我希望系统对不合理输入给出明确反馈，从而避免错误配置。
  - Given：我在填写 FIRE 配置
  - When：我输入年支出 <= 0 或 SWR <= 0（或 SWR > 20%）
  - Then：系统阻止保存并提示原因；不会产生部分写入

### 3.8 市场数据（Market Data：汇率/金价等）

#### 3.8.1 用户目标
- 手动维护关键汇率/价格，以保证多币种折算与实物资产估值可用
- 当系统检测到缺失数据时能快速补齐

#### 3.8.2 数据项（建议）
- FX：USD/CNY、HKD/CNY 等
- 商品：XAU/CNY（黄金）等
- 每条数据包含：symbol、price、更新时间、数据来源（手动/自动）

#### 3.8.3 用户故事与验收标准（Given/When/Then）
- 用户故事 MD-1：作为用户，我希望新增一条汇率数据，从而让多币种资产可折算。
  - Given：系统缺少 USD/CNY
  - When：我在市场数据管理中新增 USD/CNY=7.20 并保存
  - Then：缺失提示消失；Dashboard 与资产列表的折算金额更新
- 用户故事 MD-2：作为用户，我希望更新汇率后相关汇总即时刷新，从而保持数据最新。
  - Given：USD/CNY 当前为 7.00
  - When：我将其改为 7.20
  - Then：所有依赖该汇率的折算值刷新；并显示更新时间

### 3.9 账户（Accounts）

#### 3.9.1 用户目标
- 管理多账户（券商/银行卡/现金账户）并在资产中关联
- 在总览与资产页支持按账户汇总

#### 3.9.2 字段（建议）
- 名称（必填）
- 类型：券商/银行卡/现金/其他
- 币种（可选，若账户固定币种）
- 备注（可选）

#### 3.9.3 用户故事与验收标准（Given/When/Then）
- 用户故事 AC-1：作为用户，我希望创建一个账户，从而为资产归属提供维度。
  - Given：我在账户管理页面
  - When：我创建一个“某券商”账户
  - Then：账户列表出现该账户；资产编辑时可选择它
- 用户故事 AC-2：作为用户，我希望把资产归属到账户，从而按账户汇总查看分布。
  - Given：存在资产与账户
  - When：我在资产编辑中选择账户并保存
  - Then：资产列表显示账户信息（或在详情可见）；按账户聚合的总览数据正确

### 3.10 设置（Settings）与数据管理

#### 3.10.1 基础设置
- 本位币设置（CNY/USD 等）
- 外观设置：浅色/深色（MVP 可先只做浅色 + 预留）
- 隐私模式：隐藏金额（以“****”或模糊化展示；可一键临时显示）

#### 3.10.2 数据管理
- 导出：JSON/CSV（建议至少支持 JSON）
- 导入/恢复：从导出文件恢复数据（需要校验版本与字段）
- 清空数据：危险操作，二次确认

#### 3.10.3 用户故事与验收标准（Given/When/Then）
- 用户故事 ST-1：作为用户，我希望修改本位币，从而用习惯币种查看汇总。
  - Given：我当前本位币为 CNY
  - When：我在设置中改为 USD
  - Then：Dashboard/列表/趋势图都以 USD 作为汇总展示口径；原币种信息仍可查看
- 用户故事 ST-2：作为用户，我希望开启隐私模式，从而在公共场合不暴露金额。
  - Given：页面展示了金额
  - When：我开启隐私模式
  - Then：所有金额字段被隐藏/模糊化；趋势图可改为相对变化或隐藏数值标签；关闭后恢复
- 用户故事 ST-3：作为用户，我希望导出与导入数据，从而备份与迁移。
  - Given：我已有资产/负债/配置等数据
  - When：我导出为文件并在另一环境导入
  - Then：资产/负债/时间轴/台账/配置都被恢复；关键汇总口径一致
- 用户故事 ST-4：作为用户，我希望能清空所有数据以重新开始，但需要强提示避免误操作。
  - Given：我在数据管理中
  - When：我点击“清空数据”
  - Then：出现二次确认（可要求输入指定文本）；确认后数据被清空且不可恢复（除非有备份）

### 3.11 通用交互、校验与一致性规则

#### 3.11.1 表单校验（通用）
- 必填字段缺失时禁止提交，并明确指出缺失项
- 金额/数量/单价：
  - 默认禁止负数与 NaN/Infinity
  - 0 值需要明确语义（例如余额=0 是否允许保留记录）
  - 超大金额触发二次确认（避免误多输 0）
- 日期：
  - 到期日不能早于起息日（定期存款）
  - 交易/还款日期允许早于今天（补录历史），但需在 UI 明确显示

#### 3.11.2 币种与折算（通用）
- 所有“汇总类数字”都必须明确其币种（通常为本位币）
- 非本位币折算依赖市场数据：
  - 缺失时不应静默当作 0 而导致误导；需要显式提示“部分不可计算/缺失项”
  - 补录后应触发相关页面刷新或提示用户刷新
- 精度与舍入：
  - 货币展示遵循本位币/币种格式（千分位、符号）
  - 内部计算保留足够精度，展示时舍入

#### 3.11.3 原子性与可追溯（通用）
- 任何会影响余额/状态的操作必须满足：
  - 要么全部成功写入，要么全部失败回滚（避免部分写入）
  - 成功后写入时间轴（至少包含对象类型、动作、时间、影响金额与对比）
- 删除高风险对象（资产/负债/交易/还款记录）必须二次确认

## 4. 非功能需求（UI/UX 与系统质量）

### 4.1 设计原则
- 极简：信息密度高但不杂乱；减少装饰性元素
- 一致性：同一口径在不同页面一致呈现（汇率/币种/计入范围）
- 可解释：重要数字旁可提供“口径说明/拆分”

### 4.2 性能与稳定性
- 首页与列表：首屏加载 < 2s（本地数据库/SQLite 场景）
- 关键操作（新增/编辑/转账/还款）：需要明确成功/失败反馈，失败不应造成部分写入

### 4.3 可用性与可访问性
- 空状态、加载态、错误态完整
- 关键按钮可键盘操作；对比度满足基础可读性

### 4.4 安全与隐私
- 默认单用户本地数据（MVP），不引入账号体系
- 不在日志中输出敏感数据（金额/导出内容/个人信息）

### 4.5 数据可靠性与可恢复
- 导出文件需要带版本号与生成时间，导入时做版本兼容与字段校验
- 所有写操作需要可恢复的错误提示（失败原因、是否已写入、如何重试）
- 关键危险操作（清空/批量删除/导入覆盖）必须二次确认，并提供明确后果说明

### 4.6 兼容性与可移植性
- Web：主流桌面浏览器可用（Chrome/Safari/Edge）
- Mobile：在离线/弱网下可浏览本地数据；网络仅用于可选的市场数据更新（若未来支持）

## 5. 不在范围（Out of Scope）
- 银行/券商自动同步（OpenAPI 直连）
- 多用户协作与权限体系
- 消费记账、预算、账单识别等复杂记账功能

## 6. 技术栈建议（Tentative）
- 前端：React/Next.js + Tailwind CSS + 组件库（极简 UI）+ 图表（趋势与分布）
- 后端：Next.js API Routes
- 数据库：SQLite（开发/单用户），后续可迁移 Postgres
