# 有钱么项目代码复用重构 - 产品需求文档

## 概述
- **摘要**：重构有钱么项目的目录结构，实现代码复用，同时区分Web和移动端特定代码
- **目的**：通过提取共享代码到独立的包中，减少代码重复，提高代码可维护性，同时保持Web和移动端的特定实现分离
- **目标用户**：开发团队成员，希望通过代码复用提高开发效率和代码质量

## 目标
- 提取共享代码到独立的包中，包括类型定义、数据访问层逻辑、工具函数等
- 保持Web和移动端的特定代码分离，如数据库配置、页面组件、导航结构等
- 确保重构后的代码能够正确编译和运行
- 提高代码可维护性和可扩展性

## 非目标（超出范围）
- 不修改现有的业务逻辑
- 不添加新的功能
- 不改变现有的API接口
- 不影响现有的部署流程

## 背景与上下文
- 项目采用Monorepo结构，包含Web端和移动端两个应用
- 目前Web端和移动端有大量重复代码，如类型定义、数据访问层逻辑等
- 重复代码导致维护困难，修改一处需要同时修改多处
- 通过重构，将共享代码提取到独立的包中，实现代码复用

## 功能需求
- **FR-1**：创建共享类型定义包，包含所有数据模型的类型定义
- **FR-2**：创建共享数据访问层包，包含数据访问逻辑
- **FR-3**：创建共享工具函数包，包含通用工具函数
- **FR-4**：修改Web端项目，引用共享代码
- **FR-5**：修改移动端项目，引用共享代码
- **FR-6**：确保重构后的代码能够正确编译和运行

## 非功能需求
- **NFR-1**：代码结构清晰，易于理解和维护
- **NFR-2**：构建过程简单，能够正确处理共享代码的依赖关系
- **NFR-3**：开发体验良好，能够在本地开发时实时看到共享代码的修改

## 约束
- **技术约束**：使用TypeScript，支持Web端和移动端的不同环境
- **构建约束**：使用npm workspaces管理Monorepo结构
- **兼容性约束**：确保共享代码能够在Web端和移动端环境中正常工作

## 假设
- Web端使用Next.js框架，移动端使用Expo框架
- Web端使用better-sqlite3数据库，移动端使用expo-sqlite数据库
- 共享代码需要考虑不同环境的兼容性

## 验收标准

### AC-1：共享类型定义包
- **给定**：项目包含共享类型定义包
- **当**：Web端和移动端引用共享类型定义
- **然后**：Web端和移动端能够正确使用共享类型定义，编译无错误
- **验证**：programmatic

### AC-2：共享数据访问层包
- **给定**：项目包含共享数据访问层包
- **当**：Web端和移动端引用共享数据访问层
- **然后**：Web端和移动端能够正确使用共享数据访问层，数据操作正常
- **验证**：programmatic

### AC-3：共享工具函数包
- **给定**：项目包含共享工具函数包
- **当**：Web端和移动端引用共享工具函数
- **然后**：Web端和移动端能够正确使用共享工具函数，功能正常
- **验证**：programmatic

### AC-4：重构后的代码编译和运行
- **给定**：Web端和移动端项目引用共享代码
- **当**：构建和运行项目
- **然后**：项目能够正确编译和运行，功能与重构前一致
- **验证**：programmatic

### AC-5：代码结构清晰
- **给定**：项目目录结构重构完成
- **当**：开发人员查看项目结构
- **然后**：目录结构清晰，易于理解和维护
- **验证**：human-judgment
