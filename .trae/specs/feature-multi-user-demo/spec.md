# 多用户支持与演示模式设计规范

## 为什么 (Why)
当前应用硬编码为单用户（'default'）。为了支持多用户并允许未登录用户体验“复杂功能”（演示模式），我们需要将用户身份与应用逻辑解耦，并引入灵活的用户上下文。

## 变更内容 (What Changes)
- **重构数据访问层**: 更新所有仓库以接受 `userId` 作为参数，而不是使用硬编码的 'default'。
- **引入用户上下文**: 在前端实现 `UserContext` 以管理当前用户状态（已认证 vs. 访客/演示）。
- **演示模式实现**:
  - 创建“演示用户”策略，数据可以是只读的种子数据，或者是临时的（存储在内存/本地存储中）。
  - 添加“演示模式”的视觉指示器。
- **API 更新**: 确保所有 API 端点都遵循当前的用户上下文。

## 影响范围 (Impact)
- **受影响的规范**: 数据访问, 用户管理, API 层。
- **受影响的代码**:
  - `packages/data-access/src/repositories/*`
  - `apps/web/src/app/api/*`
  - `apps/web/src/hooks/*` (数据获取 Hooks)

## 新增需求 (ADDED Requirements)
### 需求: 多用户架构
系统必须支持多个不同的用户配置文件，每个用户都有自己独立的数据（资产、负债、设置）。

### 需求: 演示模式 (访客访问)
系统必须允许用户在不登录的情况下访问应用程序。
- **场景: 访客访问**
  - **当** 用户在没有会话的情况下访问站点时
  - **系统** 加载预先填充的样本数据的“演示模式”。
  - **并且** 显示一个横幅，指示“您正在查看演示”。
  - **并且** 写入操作（创建/更新/删除）被禁用或仅在本地生效。

## 修改的需求 (MODIFIED Requirements)
### 需求: 数据访问
所有数据访问操作（SQL 查询）必须通过 `userId` 进行范围限定。
**迁移**: 现有的 'default' 用户将被视为第一个注册用户或管理员账户。

### 需求: API 认证
API 端点必须从请求中识别用户（例如，通过 Header 或 Cookie），并将 `userId` 传递给数据层。
